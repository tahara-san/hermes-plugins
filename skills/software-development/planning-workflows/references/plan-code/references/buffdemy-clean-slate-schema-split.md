# Buffdemy clean-slate schema/content-row split checklist

Use this when a Buffdemy `/plan-code` task intentionally moves canonical content out of parent Mongo documents into dedicated child/content-row collections, or otherwise performs a clean-slate schema split with no legacy tolerance.

## Durable lessons

- **Document schemas should validate stored shape, not silently repair it.** If the plan forbids legacy document tolerance, required new metadata fields belong as required fields on `*DocumentSchema`; defaults belong only on create/input schemas.
- **Partial update schemas must be default-free.** Derive `*UpdateSchema` from a default-free base schema, then layer defaults only onto document/create schemas. Otherwise `schema.omit(...).partial()` can inject defaults such as `version`, `rank`, stats, or language cache fields into partial updates.
- **Content-row replacement must clear stale variant metadata.** If a `{parent, language}` row can switch source kinds (`machine` -> `manual`, `manual` -> `author`, etc.), an upsert that only `$set`s the new payload may leave old optional fields (`machine`, `manual`, `error`, `sourceLanguage`, `sourceContentVersion`, `sourceHash`). Prefer full replacement semantics or `$unset` all variant-only metadata before `$set`.
- **Keep parent repository transitions visibly temporary.** During an intermediate phase, parent create/update methods may be made compile-safe by stripping removed content fields, but do not mark the phase complete until the next phase actually persists/reads via content rows or the plan explicitly allows the temporary loss of content writes.
- **Parent/content writes need atomicity before review sign-off.** Once a parent helper creates or updates both metadata and a content row, make the operation transactional when no caller session is supplied, while still reusing a caller-provided `ClientSession`. Otherwise reviewers should block the phase because parent metadata can commit without its content row.
- **Preserve current API contracts during the split unless the plan changes them.** If existing API callers create/update from `papers` and do not pass a top-level `body`, derive the content-row body from validated paper bodies rather than requiring every caller to change in the schema/repository phase. Add regression tests that omit `body` to match the real caller shape.
- **Publish transitions must require ready content rows on every write path.** Helper paths, generic `update`, `findOneAndUpdate`, and direct published creates should all fail closed unless the canonical `{parent, language}` content row parses with the relevant `*ContentReadySchema`. Checking only `papers.length` or only helper-specific paths leaves publish bypasses.
- **Repository methods that return/write ready rows should parse the ready schema at the boundary.** Content-row helpers such as `upsertAuthorRow`, `updateStatus`, and batch-ready lookups may validate the broad row shape successfully even when a `status: 'ready'` row is missing required ready payload. Parse `status === 'ready'` results with the relevant `*ContentReadySchema` before returning them, and add regressions that invalid ready rows cannot be persisted or emitted by ready-row fetch helpers.
- **Soft-delete parent/content rows together on every delete-status path.** When parent documents soft-delete, propagate soft-delete to the child content rows in the same transaction. This includes direct generic parent repository calls like `update(..., { status: 'deleted' })` and `findOneAndUpdate(... $set.status = 'deleted')`, not only ownership/helper delete methods. If no caller `ClientSession` is supplied, open a transaction; if a session is supplied, reuse it. Add regressions that each public delete-status path makes `findBy*Language(parent, language)` return `null`.
- **Projected delete transitions must prefetch the full parent.** `findOneAndUpdate(..., { projection: ... })` can return a partial document without `_id`, owner, or language-cache fields. Do not use that projected result to decide which content rows to soft-delete or to run full-document schema validation. For delete-status transitions, prefetch the full matching parent inside the same session/transaction, perform the projected update, soft-delete rows using the prefetched full parent, then return the projected result when projection was requested.
- **Reject delete-with-content helper updates before writing rows.** Ownership helpers that accept both content fields and `status: 'deleted'` should fail closed instead of soft-deleting old content and then upserting a fresh `ready` row. Add regressions for delete-with-content attempts that verify parent status and existing content rows remain unchanged.
- **Mutation guards should inspect paths, not ordinary string values.** Recursive raw-update guards for protected parent content fields should look at object keys/operator paths (`$set.title`, `contentLanguageSummary.en`, etc.). Treating every string value as a possible path can falsely reject legitimate content values that happen to equal a protected field name.
- **Parse parent updates before write when the schema is the contract.** For clean-slate metadata fields such as `availableContentLanguages` and `contentLanguageSummary`, do not rely only on validating the post-write result. Pre-parse/sanitize repository `update`/`updateById` payloads with the relevant `*UpdateSchema` before calling the base repository so invalid metadata is not persisted before an error is thrown.
- **Separate replacement upserts from status-only updates.** Source-kind replacement should clear omitted variant metadata, but status-only helpers should not accidentally unset optional fields such as `sourceLanguage`, `sourceHash`, `machine`, `manual`, or `error` merely because the caller only changed `status`. Use separate update contracts or tests that prove status-only changes preserve existing variant metadata.
- **Update stale broad tests and comments immediately.** Existing schema/repository tests, helper tests, fixtures, or projection comments that still pass or describe old content fields (`title`, `body`, `papers`) become false signals after a clean split. Refresh them in the same phase so package-wide test runs do not fail later. Do not loosen production schemas just to satisfy legacy tests; update the tests to create content rows and to assert metadata-only parent shapes.
- **Content-row tests should create real parents.** Once content rows refresh parent language caches, unit tests that use orphan `new ObjectId()` parents are testing an invalid state. Create a real parent article/comment before content-row writes, and add explicit orphan/deleted-parent regressions that expect fail-closed errors instead of silently persisting rows.
- **Do not expose derived-cache writes through public parent create/update.** If an internal helper must seed `availableContentLanguages`, `contentLanguageSummary`, or similar fields after creating ready content rows, route it through a private metadata-create helper or a scoped internal path. Public repository `create`/`update` should keep rejecting caller-supplied derived cache fields.
- **Union order can silently erase structured optional subdocuments.** Avoid `z.union([z.object({}), specificSchema])` for optional "empty or configured" fields such as media controls: the empty object branch can parse first and strip valid keys to `{}` before persistence. Prefer `specificSchema.optional()`/`nullish()` plus an explicit sanitizer that omits absent values, and add regressions proving a non-empty control object survives repository parse/write/read.
- **Prefetch/update status transitions must anchor the same document.** If `findOneAndUpdate` prefetches a parent to validate publish-ready content or choose rows to soft-delete, the subsequent write must be constrained to the prefetched `_id` (and any required status/version predicate) inside the same session/transaction. Do not prefetch with a broad filter and then write with the original broad filter; a racing/broad match can publish/delete a different parent than the one validated. Add broad-filter/literal-status regressions for publish and delete.
- **Clean-slate read paths must validate emitted parents too.** Repository read helpers that bypass the base validation path (for example custom `findById`, `findByArticleId`, or `findByParentComment`) can still emit legacy parent content fields even when writes fail closed. Parse full parent results through the metadata document schema before returning them, and add read-path fail-closed tests for legacy `body`/`papers`/`title` contamination.
- **Required-index verification must include new invariants.** Creating new content-row unique indexes is not enough if startup only logs create failures and relies on `verifyRequiredIndexes` to fail hard. Add the `{parent, language}` unique indexes to required-index verification in the same phase that introduces the invariant.
- **Remove no-op publish hooks after content moves out.** When a parent publish hook's schema becomes `z.object({})` because payload validation moved to ready content rows, internal hook calls and merged payload construction add misleading complexity. Delete the internal calls or make the exported hook explicitly documented as a compatibility no-op; publish safety should live in ready-row validation.

## Review prompts should ask for these checks

Ask reviewers to verify:

1. No migration/backfill/backward-compatibility/legacy dual-read/document-tolerance code was added.
2. Stored document schemas reject missing required new fields while create schemas provide new-document defaults.
3. Update schemas do not inject defaults for omitted fields.
4. `{parent, language}` uniqueness/upsert semantics are covered, including source-kind replacement clearing stale optional metadata, and the unique indexes are included in hard required-index verification.
5. Transitional parent repository methods do not fake legacy content behavior or silently claim content persistence before the content-row write/read phase is implemented.
6. Optional nested payloads such as media controls are not parsed through an empty-object union branch that strips valid keys.
7. Publish/delete `findOneAndUpdate` paths validate and mutate the same prefetched parent under broad/racing filters.
8. Custom parent read helpers validate emitted full documents so legacy content fields fail closed on read as well as write.

## Useful targeted tests

- Stored document parse rejects documents missing newly required metadata fields.
- Create schema rejects removed canonical fields and supplies clean defaults.
- Update schema rejects removed canonical fields and returns only caller-supplied allowed fields.
- Content repository upsert replaces an existing row for the same `{parent, language}`.
- Content repository machine/manual/source replacement clears stale optional metadata.
- Batch ready-row fetch returns only `status: 'ready'` rows keyed by parent id and language.
- Media-only ready rows pass when they contain valid media but no text, while empty rows still fail.
- Non-empty media control objects survive schema parse, repository write, and readback; absent controls are omitted rather than persisted as `null` or `{}`.
- Broad-filter publish/delete `findOneAndUpdate` calls cannot validate or soft-delete content for one parent while mutating another.
- Parent read helpers fail closed when stored legacy canonical content fields are present on full read results.
- Required-index verification fails when the new `{parent, language}` unique index is missing.
